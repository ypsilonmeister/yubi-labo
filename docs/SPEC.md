## 0. 一枚まとめ（これだけは読む）

- **何を作るか:** 書字が苦手な7歳児のための、Webブラウザで動くトレーニングゲーム集（3モード）。鉛筆・紙を使わず、タッチ／マウス／Webカメラの指先トラッキングで遊ぶ。
- **なぜ作るか:** 対象児は「目と手の協応」と「文字を音に変える処理（デコーディング）」が著しく弱く、「形の理解・言語理解・順序記憶」が非常に強い。弱点を、強みを使って、失敗体験なしに鍛えるため。
- **絶対原則（§2）:** ❶失敗・減点・ゲームオーバーを作らない ❷UIに文字を使わない（音声とアイコンのみ） ❸急かさない ❹成果は記録して翌日見せる ❺1セッション5分で気持ちよく終わる。
- **実装順序（§10）:** P0基盤 → P1モードA（迷路・タッチ入力）→ P2モードB（漢字パズル）→ P3ハンドトラッキング → P4モードC（もじさがし）→ P5保護者画面。**P1完了時点で一度ユーザー（保護者）に見せること。**

## 1. 背景・目的・対象ユーザー

### 1.1 目的

対象児（7歳・小学2年生、以下「本人」）の以下2つの弱点を、ゲームとして楽しく反復できる形で補強する。

1. **視覚運動統合（目と手の協応）:** 見た通りに手をコントロールして動かす力。
2. **デコーディング（文字→音の変換の自動化）:** 文字を見て瞬時に読みに変換する力。

同時に、**「自分は上手にできる」という成功体験を蓄積させる**こと自体が目的の半分である。トレーニング効率よりも「本人が自分から遊びたがるか」を優先する。

### 1.2 対象ユーザーの認知プロファイル（2026年6月実施の視覚認知発達検査に基づく）

設計判断の根拠となるため、実装者も必ず把握すること。

| 領域 | 検査結果 | 水準 | 設計への含意 |
|---|---|---|---|
| 眼と手の協応 | DTVP-3 標準得点 **4**（2パーセンタイル） | **著しく低い** | モードAの主ターゲット。難易度の初期値は極めて易しく |
| 視覚運動統合指数 | **76**（5パーセンタイル） | 低い | 同上 |
| 模写 | 標準得点 8 | 平均域の下限 | なぞり（トレース）系は失敗しにくい入口になる |
| 図と地（探索） | 標準得点 12 | 平均上 | モードCの探索操作は得意。ここは頼ってよい |
| 視覚形態完成 | 標準得点 14（91パーセンタイル) | 優れている | パーツから全体を構成する遊びが向く |
| 形の恒常性 | 標準得点 **17**（99パーセンタイル） | 非常に優れている | 形の回転・変形を使った演出は理解できる |

| 言語性処理速度（ひらがな呼称） | DEM 標準得点 **55**（1パーセンタイル未満） | **最重要の弱点** | モードCの主ターゲット。文字→音の結合を毎回強化する |
| 基本的な眼球運動（追従・衝動性） | 問題なし | 正常 | **眼球運動そのものは鍛える対象ではない**。視線訓練を狙った仕様にしないこと |
| 教示理解（言語） | 非常にスムーズ | **強み** | **すべての説明は「話し言葉」で行う。** 本人は視覚的に真似るのが苦手な代わりに、言葉で手順を説明されると理解できる「言語優位」タイプ |
| 姿勢保持・粗大運動 | 長時間の姿勢保持が困難 | 弱い | 腕を上げ続ける操作を長時間要求しない（§4.2.6） |
| 注意集中 | 課題中は集中良好 | 強み | 1課題は短く区切れば最後までやり切れる |

### 1.3 性格・情緒面で設計上考慮すべきこと

- **否定的なフィードバックが長く残る特性**がある（本人は「嫌な言葉は頭の中で雲みたいに回り続ける」と自己表現している）。❌マーク・ブザー音・「ざんねん」等は一度も見せてはならない。
- **楽しかった記憶はすぐ忘れる**特性も本人が自覚している。→ 成果を自動記録し、次回起動時に見せ返す機能（§4.6）は本アプリの中核機能であり、省略不可。
- **焦り・時間切迫がかんしゃくのトリガー**になる。制限時間による失敗は作らない。
- 興味関心: **植物、DNA・分子、宇宙、実験**。テーマ・演出はここに寄せる（§5.5, §6.6）。
- 「字の練習」と呼ぶと書字への嫌悪感が転移するリスクがある。アプリ内でも保護者向け文言でも「書字練習」とは表現せず「ゆびのゲーム」「実験」と呼ぶ。

## 2. 設計原則（GUARDRAILS — 実装中に迷ったらここに戻る）

以下は**変更禁止**。機能追加・省略の判断で矛盾が生じた場合、常に本節が優先される。

1. **NO-FAIL原則:** ゲームオーバー・スコア減点・❌表示・エラー音・残機・制限時間切れによる失敗、のすべてを実装しない。ミスに相当する事象（はみ出し等）は「緩やかな視覚変化」のみで伝える（各モードで定義）。
2. **テキストレスUI:** 子ども向け画面に文字（ひらがな含む）を表示しない。ナビゲーションはアイコン＋音声ガイダンス。例外は §6（漢字・ひらがな自体が教材であるモードB/Cのゲーム内容物）と保護者画面。
3. **音声ガイドは「完全な言語指示」:** 本人は言語優位。音声は「これをこうして」のような指示語ではなく、「みどりの まるを ゆびで さわってね」のように対象・動作を明示的に言語化する（台本: 付録B）。
4. **自己比較のみ:** 他者比較・ランキング・平均値との比較を一切表示しない。比較対象は常に「過去の本人」。
5. **セッションは短く、気持ちよく終わる:** 1セッション標準5分（設定可能: 3〜10分）。終了は視覚タイマー（円グラフ状に減る）で予告し、終了時は必ず「ハイライト再生→ほめる音声→また明日」の順で閉じる。「もう1回」の連打で延長できる上限は+2分まで。
6. **急かさない:** すべての課題は自己ペース。速度計測は裏で記録するが、本人にカウントダウンやタイマー音を聞かせるのはモードCの「チャレンジモード」（本人が自分で選んだ時のみ、§7.4）に限る。
7. **入力機器のせいにさせない:** ハンドトラッキングの検出が不安定なときは、本人の操作ミスに見える形で表出させず、「カメラのマーク＋『カメラが みえてないみたい』音声」で機器側の問題であることを明示する（§4.2.5）。
8. **ローカル完結:** カメラ映像・記録データを外部送信しない。すべての永続化はローカル（§4.5）。アナリティクス・外部API送信は実装しない。

## 4. 共通基盤仕様（P0）

### 4.1 入力抽象化レイヤ `PointerSource`

全ゲームは入力を以下のインターフェース経由でのみ受け取る。**ゲームロジックはマウス／タッチ／ハンドトラッキングを区別してはならない。**

```ts
interface PointerState {
  x: number; y: number;        // 論理座標(0-1280, 0-800)
  active: boolean;             // 「押している/掴んでいる」状態
  confidence: number;          // 0-1。マウス/タッチは常に1
  timestamp: number;
}
interface PointerSource {
  readonly kind: 'mouse' | 'touch' | 'hand';
  subscribe(cb: (s: PointerState) => void): () => void;
  start(): Promise<void>;
  stop(): void;
}
```

- `active` の意味論: マウス=ボタン押下、タッチ=接触、ハンド=ピンチ or ドウェル（§4.2.4）。
- 入力ソースはホーム画面のアイコンで切替（🖐カメラ / 👆タッチ）。カメラが無い環境ではハンドのアイコンを表示しない。

### 4.2 ハンドトラッキング仕様（P3で実装）

#### 4.2.1 検出

- MediaPipe `HandLandmarker`、`numHands: 1`、`runningMode: VIDEO`。
- 使用ランドマーク: `8`(INDEX_FINGER_TIP)=カーソル、`4`(THUMB_TIP)=ピンチ判定、`0`(WRIST)と`9`(MIDDLE_FINGER_MCP)=手のスケール算出、`11`付近(胸中心推定には Pose は使わず、後述の相対補正で代替)。
- カメラは前面ユーザーの上半身が映る想定。**映像はcanvasに表示しない**（自分の顔が映ると気が散る）。代わりに画面隅に小さな手アイコンで検出状態を示す（検出中=緑、ロスト=グレー）。

#### 4.2.2 座標変換

- 初回のみ簡易キャリブレーション:「みぎうえの ほしを ゆびで さしてね」→「ひだりしたの ほしを さしてね」の2点＋中央1点で、カメラ正規化座標→論理座標のアフィン変換を決定。結果は保存し、次回以降スキップ可（ホームの🔧アイコンから再実行可能）。
- ホモグラフィ4点方式でもよいが、7歳児が4点静止するのは負荷が高いため3点以内とする。

#### 4.2.3 平滑化（必須）

One Euro Filter を x, y 各軸に適用。初期パラメータ:

```
freq = 実測フレームレート, minCutoff = 1.2, beta = 0.01, dCutoff = 1.0
```

- 目視でカーソルの震えが線の判定幅（§5.3）の1/3以下になるまで `minCutoff` を下げて調整（下げすぎると遅延が出て「思った所に来ない」ため、遅延体感<100msを守る）。
- `confidence < 0.5` のフレームは座標更新せず前回値を保持。

#### 4.2.4 「押す/掴む」ジェスチャー

優先度順に2方式を実装し、設定で切替可能にする:

1. **ドウェル方式（デフォルト）:** カーソルが対象の上に **800ms** 静止（移動半径 < 30px）で `active=true`。掴んでいる間は移動自由、対象外で **500ms** 静止 or 手のロストで `active=false`（=ドロップ）。進捗はカーソル周囲のリング（円弧が満ちる）で表示。
2. **ピンチ方式:** 親指先(4)と人差し指先(8)の距離が手のスケール（WRIST–MIDDLE_MCP距離）の **0.35倍未満**でピンチON、**0.55倍超**でOFF（ヒステリシス必須）。

> 理由: 対象児は協調運動が弱く、ピンチの誤爆がフラストレーションになるため、失敗しようがないドウェルを標準とする。

#### 4.2.5 トラッキングロスト時の振る舞い（原則7）

- 手を500ms以上検出できない → ゲームを**自動ポーズ**し、手アイコンをグレー表示＋音声「カメラから てが みえなくなっちゃった。カメラの まえで ゆびを だしてね」。
- 復帰したら「みつけた！」の音声とともに3秒のカウント演出後に再開。**ロスト中の時間・操作は一切ペナルティにしない。**

#### 4.2.6 疲労対策（姿勢保持が弱いことへの配慮）

- ハンド入力時、**連続使用90秒で自動的に「ひとやすみ」演出**（10秒、腕を下ろすよう促す音声:「いちど うでを おろして ぶらぶらして いいよ」）を挟む。ゲーム状態は保持。
- ハンド入力のゲームは1プレイ単位を60秒以内で設計する（各モードのレベル長で担保）。

### 4.3 音声ガイド `AudioGuide`

- `speak(key: string)`: 付録Bのキーを再生。再生中の重複呼び出しはキューせず**置き換え**（テンポ優先）。
- 同一キーの連続再生は3秒以内なら抑止（うるさくしない）。
- 音量設定は保護者画面のみに置く。
- **すべてのインタラクション可能要素は、初回表示時に1度だけ音声で説明される**（例: ホーム初回起動時「あそびたい ゲームを ゆびで えらんでね」）。

### 4.4 UIシェル

#### 4.4.1 画面遷移

```
[起動] → (初回のみ 名前ではなくアバター選択: 🌱/🧬/🪐 から1つ)
  → [きのうのハイライト] (§4.6, 記録がある場合のみ, スキップ可)
  → [ホーム] → 各ゲーム → [セッション終了画面] → ホーム
```

#### 4.4.2 ホーム画面

- 大きなアイコンカード3枚（モードA=🌱迷路、B=🧬かんじ工場、C=🔍もじさがし）。文字ラベルなし。ホバー/フォーカスで音声によるモード名読み上げ。
- 右上: 入力切替（👆/🖐）、🔧（キャリブレーション）、⚙（保護者画面への入口。**3秒長押し**で開く=子どもの誤操作防止）。
- 左上: セッション残り時間の視覚タイマー（円が欠けていく。数字なし）。

#### 4.4.3 セッションタイマーと終了フロー（原則5）

- セッション長: 保護者設定（デフォルト5分）。ゲームプレイ中のみ減算（ハイライト再生・ひとやすみ中は停止）。
- 残り60秒: タイマーが柔らかく点滅＋音声「そろそろ おわりの じかんが ちかづいてるよ」。
- 0秒: 現在のレベルはキリの良い所まで続行可（強制中断しない）→ レベル完了時に終了画面へ。
- 終了画面: 今日のハイライト自動再生（§4.6）→ 音声「きょうも たくさん じっけんできたね。つづきは あしたの おたのしみ！」→ 🌙アイコンでアプリ終了状態（操作ロック）。
- 「+2分」延長アイコンは1回だけ表示（原則5）。

### 4.5 データ永続化

IndexedDB。ストア構成:

```ts
// store: sessions
interface SessionRecord {
  id: string;             // uuid
  startedAt: number; endedAt: number;
  inputKind: 'mouse'|'touch'|'hand';
  plays: PlayRecord[];
}
// 1レベル/1問ごとの記録
interface PlayRecord {
  game: 'maze'|'kanji'|'moji';
  levelId: string;
  startedAt: number; durationMs: number;
  completed: boolean;              // 途中終了=false（失敗ではない）
  metrics: MazeMetrics | KanjiMetrics | MojiMetrics; // 各モードで定義
  replay?: ReplaySample[];         // ハイライト用軌跡（間引き済み）
}
interface ReplaySample { t: number; x: number; y: number; active: boolean; }
// store: progress  … ゲームごとの解放状況・図鑑コレクション
// store: settings  … セッション長、入力方式、キャリブレーション行列、音量
```

- `replay` は 30ms 間隔に間引いて保存。1プレイ最大2000サンプルで打ち切り。
- **エクスポート:** 保護者画面から全データをJSONダウンロード可能にする（療育機関との共有・バックアップ用）。インポートも実装。

### 4.6 ハイライト機能「きのうのきみ」（省略不可）

> 背景: 本人は「楽しかった記憶がすぐ消える」特性を自覚している。成功体験をアプリが外部記憶として保持し、見せ返すことが本アプリの核心的価値。

- **記録:** 各プレイの `replay` と成果（完成した漢字、クリアした迷路の軌跡など）を保存。
- **選定:** セッション終了時と翌回起動時に「ハイライト」を1〜3件自動選定。選定基準（優先順）: ①初めて完成した漢字 ②自己ベストを更新した迷路（はみ出し率低下 or 通路レベル更新）③最後にクリアしたレベル。
- **再生:** 実プレイの軌跡を2倍速でアニメーション再生。軌跡はキラキラのパーティクルで描画し、完成時にファンファーレ（柔らかい音）＋音声「これ、きのうの きみだよ。じょうずだったねえ」。
- 再生は10秒以内。スキップ用アイコン（▶▶）あり。

### 4.7 保護者画面（P5）

- 入口: ホームの⚙3秒長押し。文字使用OK（日本語）。
- 内容: ①セッション履歴一覧 ②モード別の推移グラフ（迷路: はみ出し率・到達レベル / 漢字: 習得数・定着率 / もじさがし: 平均反応時間）③設定（セッション長、入力方式、音量、モードC チャレンジ許可）④データのエクスポート/インポート ⑤漢字出題範囲の選択（1年生漢字の中からチェックボックス）。
- グラフは本人には見せない前提のUI（数値・平均線を含んでよいが、他者比較データは存在しない）。

## 5. モードA「ねっこの迷路」（運筆コントロール）

### 5.1 目的とデザインテーマ

- **狙う機能:** 目と手の協応 ——「思った通りに動かす」「狙った所で止める」のブレーキコントロール。
- **テーマ:** 植物の根っこ。プレイヤーの指は「根の先端」。通路=土の中の柔らかい道。ゴール=水や栄養（💧✨）。到達すると画面上部の植物が少し育つ（レベルクリアごとに成長し、コレクション化 §5.6）。

### 5.2 基本ルール

1. スタート地点（種のアイコン）にカーソルを合わせると音声「ここから ねっこを のばそう。みずいろの みちに そって ゆっくり すすんでね」→ 根の描画開始。
2. カーソル移動に沿って根（茶色の線＋先端に小さな根毛アニメ）が伸びる。
3. 通路内を進む: 根は太く健康的な描画。**通路からはみ出す: 根が細く・色が薄くなるだけ**（原則1。音・停止・巻き戻しなし）。通路に戻ると2秒かけて太さが回復。
4. 途中の「ストップポイント」（🔴赤いお日さまマーク）に到達したら、その場で静止ホールド（§5.3）。成立すると🔴→🟢に変わり音声「ピタッと とまれたね！」。
5. ゴール到達でクリア演出（花が咲く/実がなる、3秒）。

### 5.3 判定パラメータ

| 項目 | 値 | 備考 |
|---|---|---|
| 通路内判定 | カーソル中心が通路ポリライン±通路半幅以内 | |
| はみ出し記録 | 10ms単位で通路外時間を積算 → `outRatio = 通路外時間/総時間` | 本人には見せない。保護者グラフ用 |
| ホールド成立 | 半径 **45px**（ハンド入力時は **60px**）内に **3.0秒** 滞在 | 進捗リング表示。途中で出たらリングが**ゆっくり減る**（リセットしない） |
| 最低通路幅 | タッチ/マウス: 40px、ハンド: 70px | これ未満のレベルは入力方式に応じて自動的に幅を広げる |

### 5.4 レベル設計

`src/data/mazes.json` に定義。スキーマ:

```ts
interface MazeLevel {
  id: string;              // "maze-01" 等
  path: [number, number][];// 通路中心のポリライン（論理座標）
  width: number;           // 通路の全幅(px)
  stops: number[];         // ストップポイントの位置（pathに沿った距離の割合 0-1）
  theme: 'soil'|'rock'|'water'; // 背景バリエーション
}
```

初期収録20レベルの難易度カーブ（実装者がこの表に沿って作成する）:

| レベル帯 | 通路幅 | 形状 | ストップ数 |
|---|---|---|---|
| 1–4 | 120px | ほぼ直線→大きなカーブ1つ | 0–1 |
| 5–8 | 100px | S字、L字 | 1 |
| 9–12 | 80px | ジグザグ、渦巻きの入口 | 1–2 |
| 13–16 | 60px | 渦巻き（ぐるぐる迷路）、狭まる通路 | 2 |
| 17–20 | 40–50px | 渦巻き＋折返し複合 | 2–3 |

- **レベル解放:** クリアで次が解放。`outRatio > 0.35` のクリアが2回続いたら、次レベルの幅を+20px する自動救済（本人には通知しない）。
- **1レベルの想定所要:** 20〜60秒（§4.2.6の疲労対策と整合）。

### 5.5 フィードバック仕様

- クリア時音声のバリエーション（付録B `maze.clear.*`）をランダム再生。**内容は結果でなくプロセスを褒める**（「さいごまで ゆっくり すすめたね」「とまるのが じょうずに なってきた」）。
- はみ出しに言及する音声は存在しない（禁止）。

### 5.6 コレクション「はたけ」

- クリアごとに植物が1段階成長（種→芽→葉→花→実、5段階）。20レベル=4株の植物が完成。
- ホームから🌱アイコンで「はたけ」を眺められる（操作なし、癒し画面）。

## 6. モードB「かんじ工場」（マルチセンソリー漢字パズル）

### 6.1 目的とデザインテーマ

- **狙う機能:** 漢字パーツの構成理解（強み）×順序記憶（強み）×音との結合（弱点）。書かずに漢字を定着させる。
- **テーマ:** 分子合成の実験室。漢字パーツ=原子、完成した漢字=分子。合体時は原子が結合するエフェクト（パーツ同士が引き合い、結合音「ポンッ」＋光）。

### 6.2 1問の流れ（シーケンス提示 → 組み立て → 多感覚定着）

1. **ターゲット提示:** 完成形の漢字が画面中央に2秒表示され、音声で読みと意味「これは『やすむ』。ひとが きのよこで やすんでる じ だよ」（成り立ちストーリー、付録データに含む）。
2. **シーケンス提示（重要・本人の順序記憶の強みを使う）:** 完成形が分解され、パーツが**1つずつ順番に**正しい位置へ飛んでいくアニメーション（各1秒、音声「まず にんべん。つぎに き」）。
3. **組み立て:** パーツがシャッフルされて画面下部に並ぶ。本人がパーツを掴み（ドラッグ/ドウェル）、枠内のドロップゾーンへ運ぶ。
   - ドロップゾーンは**淡いシルエットで表示**（レベルが上がるとシルエットが薄くなる: §6.4）。
   - 正しいゾーンの近く（中心から80px以内）で離すと**吸い付いて**ハマる（スナップ）。
   - 違うゾーンに置いた場合: パーツが「ぷるぷる」と首を振って**ふわりと元の位置に戻る**だけ。音声もエフェクトも出さない（原則1）。3回続いたら音声ヒント「『き』は みぎがわ だよ」。
4. **完成演出（多感覚の同時発火）:** 全パーツ結合 → 漢字全体が光る → **読み上げ音声＋その漢字の意味アニメーション**（例:「休」→ 木の横で人が座って休む2秒アニメ）→ 「『やすむ』、ゲットだね！」
5. **想起チェック（3問に1回）:** 直前にやった漢字について「『やすむ』は どれだった？」と3択（完成形の漢字3つを表示）→ タップ。正解で図鑑に✨マーク。不正解でも「こたえは これ。もういっかい つくってみよう」→ 同じ漢字の組み立てをもう1度（ペナルティ扱いにしない）。

### 6.3 漢字データスキーマ

`src/data/kanji.json`:

```ts
interface KanjiEntry {
  id: string;            // "kanji-hayashi"
  char: string;          // "林"
  reading: string;       // 音声用よみ "はやし"
  meaningAudioKey: string; // 付録B参照
  storyText: string;     // 成り立ちの言語化（音声原稿）
  grade: 1 | 2;
  parts: KanjiPart[];    // 組み立て順に並べる（=シーケンス提示順）
}
interface KanjiPart {
  glyph: string;         // 表示用（単体で表示可能な字形。例 "木"。偏など単体字がない場合はSVGパスを指定）
  svgPath?: string;      // glyphで表現できないパーツ用
  zone: { x: number; y: number; w: number; h: number }; // 完成枠内の正規化位置(0-1)
  spokenName: string;    // "にんべん" "き" など音声ヒント用
}
```

**初期収録（P2で最低この12字を実装。以後保護者画面から範囲拡張）:**

| 漢字 | パーツ分解（順） | 選定理由 |
|---|---|---|
| 林 | 木＋木 | 2パーツ・同一形の最易問 |
| 森 | 木＋木＋木 | 3パーツ・積み上げ |
| 休 | 亻＋木 | 成り立ちストーリーが明快 |
| 本 | 木＋一 | 既知パーツ＋1画 |
| 田 | 口＋十 | 幾何学的 |
| 男 | 田＋力 | 既習パーツ再利用 |
| 音 | 立＋日 | 上下構成 |
| 早 | 日＋十 | 上下構成 |
| 明 | 日＋月 | 左右構成・意味アニメが作りやすい（日と月で明るい） |
| 岩 | 山＋石 | 本人の興味（地質）に接続 |
| 花 | 艹＋化(亻＋匕) | 本人の興味（植物）に接続 |
| 草 | 艹＋早 | 既習「早」の再利用=シーケンス記憶の転移 |

- パーツの再利用関係（早→草 等）を意図的に含めること。「しってる パーツだ！」体験が定着を促す。
- 2年生配当漢字への拡張はデータ追加のみで可能な構造にする。

### 6.4 難易度段階（漢字ごとに独立して進行）

| 段階 | シーケンス提示 | ドロップゾーン | 想起チェック |
|---|---|---|---|
| Lv1 | あり（毎回） | 濃いシルエット | なし |
| Lv2 | あり | 薄いシルエット | あり（3択） |
| Lv3 | **なし**（音声で読みだけ提示→記憶で組む） | 枠のみ（シルエットなし） | あり |

- 同じ漢字をLv3でノーヒント完成 → 図鑑で「マスター」（金の分子マーク）。
- 出題アルゴリズム: 未習1 : 復習2 の割合で混ぜる（間隔反復の簡易版: 最終正解から 1日→3日→7日 後に再出題キューへ）。

### 6.5 図鑑「ぶんしずかん」

- 完成した漢字が分子模型風のカードで並ぶ。タップでいつでも完成演出をリプレイ可能。
- カード数がそのまま学習量の可視化になる（数字を出さずに「増える棚」で見せる）。

### 6.6 意味アニメーション

- 各漢字2秒のシンプルなアニメ（SVG/Canvas）。実装コスト削減のため「静止イラスト2枚のクロスフェード＋効果音」で可とする。イラストは絵文字組み合わせで代用可（例: 明 = 🌞＋🌙 が並んで画面が明るくなる）。

## 7. モードC「もじさがし」（デコーディング自動化）

### 7.1 目的（重要: 眼球運動訓練ではない）

- **狙う機能:** 文字→音の変換（デコーディング）の自動化。検査上、眼球運動そのものは正常で、弱いのは「見た文字を音にする速度」。したがって本モードの設計主眼は**「音と文字の結合を1タップごとに強化する」**ことであり、視線ジャンプの訓練を狙った仕様（視標の高速点滅等）は実装しない。
- 副次的に、得意な「図と地」（視覚探索）を成功体験の土台として使う。

### 7.2 基本ルール（ききとりモード=デフォルト）

1. 画面に文字カード（ひらがな）が **4〜9枚**、不規則な位置に表示される（重ならない、最小間隔100px）。
2. 音声が出題:「『ね』は どこかな？」
3. 本人が探してタップ/ポイント。
   - **正解:** カードが弾けて音声「ね！」（文字の音を即時再生=結合強化）＋柔らかい効果音。
   - **他のカードをタップ:** そのカードの音を読み上げるだけ「それは『わ』だね。『ね』を さがしてみよう」（**誤答扱いにしない**。触った文字の音も学習機会にする）。
4. 8問で1セット。セット終了で星のカードが1枚たまる（コレクション）。

### 7.3 出題設計

- **文字セット段階:**
  1. 清音・形の弁別が易しい組（あ行・か行など形の差が大きい組合せ）
  2. 清音46字全体
  3. 形が紛らわしいペアを意図的に混ぜる（ね/わ/れ、は/ほ、ぬ/め、る/ろ、き/さ）
  4. 濁音・半濁音
  5. カタカナ（同手順）
  6. 既習漢字（モードBで「マスター」した字を出題対象に自動追加。**モードB↔C連携**）
- 適応制御: 直近20問の文字別正答率と反応時間を記録。正答率<60%の文字は出題頻度を上げ、選択肢数を4枚に減らす。>90%かつ反応が速い文字は間引く。
- `MojiMetrics = { char: string; correct: boolean; reactionMs: number; distractors: string[] }`

### 7.4 チャレンジモード（本人が選んだ時だけ）

- ホームからではなく、ききとりモードを3セット終えると⚡アイコンが出現し、**本人がタップした場合のみ**発動。
- 内容: 60秒で何枚見つけられるか（自己ベストのみ表示。星の数で表現し数字は出さない）。残り時間は円タイマーで表示、**カウントダウン音声・点滅・警告音は無し**。時間切れは「しゅうりょう〜！ きょうは ほし◯こ！」と成果側だけ言う。
- 保護者画面でチャレンジモード自体を無効化できる（焦りが強い時期はオフにする想定）。

### 7.5 よみあげモード（将来拡張・P6以降、実装は任意）

- 逆方向（文字を見て本人が発声→Web Speech API の音声認識で判定）。誤認識が本人の失敗に見えるリスクが高いため、v1では**実装しない**。仕様枠だけ確保。

## 8. 難易度自動調整の共通原則

- すべてのモードで「連続クリア→静かに難化 / 苦戦の兆候→静かに易化」を行い、**難易度変更を本人に通知しない**。
- 苦戦の兆候の定義: ①同一レベル/問で90秒経過 ②モードAで `outRatio>0.35` が2回連続 ③モードCで同一文字3回連続の探索に20秒超。
- 易化の手段はモードごとに定義済み（A: 通路幅+20px / B: シルエット濃度を1段階戻す・音声ヒント即時化 / C: カード枚数-2）。
- **90秒以上の停滞時:** 音声「ちょっと むずかしいね。いっしょに やってみよう」→ 半自動化（A: ゴールまで光る誘導線 / B: 残りパーツが1つずつ自動で吸い込まれるのを見る / C: 正解カードがゆっくり点滅）。**「クリアできなかった」状態でその日を終えさせない。**

## 9. 非機能要件

| 項目 | 要件 |
|---|---|
| プライバシー | カメラ映像・音声・記録の外部送信ゼロ。ネットワーク通信はMediaPipeモデルの初回取得のみ（可能ならモデルをバンドルしローカル配信） |
| 性能 | ゲーム中60fps目標、30fps下限。トラッキング処理はゲーム描画と別（`requestVideoFrameCallback` or Worker） |
| 起動 | 起動→ホーム表示まで5秒以内（子どもの興味が持続する限界） |
| 音声 | 初回タップ前にAudioContextを resume するユーザージェスチャーを設ける（ブラウザの自動再生制限対策。起動画面の「はじめる」🌟タップがこれを兼ねる） |
| エラー処理 | カメラ拒否/不在→タッチモードへ自動フォールバック＋保護者向けトースト。IndexedDB失敗→メモリ動作で続行しセッション終了時に警告 |
| 対応解像度 | 1024×768〜。縦持ちタブレットでは回転を促す画面（アイコンのみ） |
| フォント | 教科書体系のWebフォント（UDデジタル教科書体が理想、なければ「Klee One」等の教科書体系Googleフォント）。**明朝・ゴシックの「き」「さ」等は手書き字形と異なるため教科書体必須** |

## 10. 実装フェーズと受け入れ基準

| フェーズ | 内容 | 受け入れ基準（すべて満たすこと） |
|---|---|---|
| **P0** 基盤 | シェル、PointerSource(mouse/touch)、AudioGuide(TTS)、IndexedDB、視覚タイマー、終了フロー | ホーム→ダミーゲーム→終了画面が音声ガイド付きで一巡する。文字が子ども画面に1つも無い |
| **P1** モードA | 迷路20レベル、はたけ、ハイライト記録・再生 | タッチ/マウスで20レベル遊べる。はみ出しても不快な表示・音が一切出ない。翌回起動で「きのうのきみ」が再生される |
| **P2** モードB | 漢字12字、3段階難易度、図鑑、想起チェック | 12字すべてで提示→組立→完成演出→図鑑登録が動く。誤配置で否定的フィードバックが出ない |
| **P3** ハンド入力 | MediaPipe、OneEuro、キャリブレーション、ドウェル/ピンチ、ロスト処理、疲労休憩 | ハンドでモードA Lv1–8とモードBが完走可能。ロスト時にゲームが失敗しない。カーソル遅延が体感100ms未満 |
| **P4** モードC | ききとりモード、適応出題、チャレンジ | ひらがな段階1–3が動く。誤タップで否定的フィードバックが出ない |
| **P5** 保護者画面 | 履歴、グラフ、設定、エクスポート | JSONエクスポート→インポートで完全復元できる |

- **P1完了時に必ず一度、保護者にプロトタイプを提示すること**（2026年9月5日に療育機関でのトレーニング開始予定があり、そこで専門家に見せてフィードバックを得る計画がある）。
- 各フェーズはそれ単体で子どもに渡せる品質にする（未実装モードのアイコンはホームに表示しない）。

## 11. 実装者への裁量範囲

**変えてよいもの:** 技術スタックの同等品への置換、レベルデザインの調整、演出のクオリティアップ、コード構成。
**変えてはならないもの:** §2の8原則、§4.6ハイライト機能の存在、モードCの狙い（デコーディングであって眼球運動でないこと）、モードBのシーケンス提示（順序記憶を使う核アイデア）、音声=完全な言語指示という方針。
**迷ったら:** 「7歳の本人が一人で遊んで、嫌な気持ちになる瞬間がゼロか？」で判断する。

## 12. v1.1 改訂（2026-07）— 難易度天井の引き上げ

### 12.1 改訂の背景

v1.0 の全コンテンツ（P0–P5）実装後、本人がタッチ/マウス入力で**全モードを余裕で踏破**した。v1.0 の難易度は検査結果の最悪ケース（眼と手の協応 2%ile）に合わせた保守的設定であり、実際の遊びの中でのパフォーマンスはそれを大きく上回った。本人は PhET Interactive Simulations で自発的に遊ぶレベルにあり、「パラメータをいじって因果を発見する自由実験」を好む。v1.1 は訓練ターゲット（視覚運動統合・デコーディング）を維持したまま、難易度の天井と伸びしろを本人の実力レベルへ引き上げる。

> ハンドトラッキング入力は本改訂時点で本人未評価。ハンド用の閾値・クランプ（§4.2、§5.3）は**別軸としてすべて維持**する。

### 12.2 ガードレール改訂（§2-2 のみ変更、他は不変）

- **§2-2 テキストレスUI → 緩和:** 子ども向け画面へのひらがな・漢字の**補助表示を許可**する（例: ホームカードのラベル、図鑑カードの読み、もじさがしの綴りスロット）。デコーディング練習の機会をUI側にも作る意図。ただし**音声ガイドが主、文字は従**の原則を守り、文字がないと操作できないUIは引き続き禁止。
- **§2 の他7原則（NO-FAIL・完全な言語指示・自己比較のみ・気持ちよく終わる・急かさない・機器のせいにさせない・ローカル完結）と付録Bの禁止語彙は一切変更しない。**難易度をどれだけ上げても「7歳の本人が一人で遊んで嫌な気持ちになる瞬間がゼロか？」（§11）が最終判定基準であることは不変。

### 12.3 §8 への追記 — 上方向の適応制御

v1.0 の適応は易化方向のみだった。v1.1 では全モードに**昇格ロジック**を追加する。難易度変更を本人に通知しない原則（§8）は上方向にも適用する。

- **モードA:** progress に `maze.skill`（0–5）を永続化。クリーンクリア（`outRatio < 0.10`・救済なし・全ストップ完了）で +1、`outRatio > 0.35` で -1。実効通路幅 = `max(width - skill*4 + reliefBonus, 入力別最小幅)`。skill=5 でのクリーンクリア2連続時のみレベルを+2スキップ。
- **モードB:** Lv4 の新設（12.5）。
- **モードC:** ステージ追加（12.6）と従来のカード枚数昇格の継続。

### 12.4 モードA 拡張 — レベル21–44 と新メカニクス

`scripts/generate-mazes.mjs`（決定論ジェネレータ、シード固定）で生成し JSON をコミットする。難易度バンド:

| レベル帯 | 通路幅 | ホールド半径 | 形状 | ストップ数 | 特殊 |
|---|---|---|---|---|---|
| 21–26 | 36px | 40px | 渦巻き複合・長いS | 2–3 | — |
| 27–32 | 32px | 36px | 折返し密集 | 3 | branch（行き止まり枝）1本 |
| 33–38 | 28px | 32px | 渦＋折返し | 3–4 | memory 2レベル |
| 39–44 | 24px | 30px | 複合 | 3–4 | memory + branch |

- **branch:** 主経路とは別ポリラインの行き止まり枝。通路内判定には合算するが、進捗・ゴール・ストップは主経路のみから計算。枝の探索中も根は痩せない（行き止まりは失敗ではなく探索）。枝の終端はゴールから150px以上離す。
- **memory:** 開始前は通路をフル表示（自己ペースのプレビュー）。トレース開始後に通路がゆっくり薄くなる（alpha 0.28→0.04、完全消去はしない）。強みの順序記憶で「道を覚えて進む」。専用救済 = エコー再表示（health<0.6 が1.5秒続いたら現在位置周辺の道がほんのり再点灯）。memory レベルの outRatio は relief/skill 判定から除外。
- スキーマ追加: `holdRadius?`（レベル別ホールド半径、ハンド時は §5.3 の値で下支え）、`branches?`、`memory?: {previewGraceMs, echoMs}`。
- はたけ（§5.6）はレベル総数から株数を導出する（44レベル→9株）。

### 12.5 モードB 拡張 — 収録字の拡大と Lv4

- **収録字:** 12字 → 36字以上へ波状拡張。選定基準: ①全パーツが単独表示可能な1グリフ（鳥・馬・魚などの一体字、光・風・谷などグリフ分解不能字は収録しない） ②既習パーツの再利用優先（順序記憶の転移） ③興味領域（植物・宇宙・地学・実験）優先。`scripts/kanji-source.mjs` + `generate-kanji.mjs` でレイアウト原型（LR/TB/KANMURI/ENCLOSE等）から zone を半自動生成。
- **Lv4（新設）:** 完成形を見せず読み音声のみで出題 + 正解パーツに**ダミーパーツ2枚**を混ぜる。ダミーはどのゾーンにもスナップせず、ふわりとトレイに戻るだけ（否定表示なし）。Lv4 をノーヒント完成で `lv4Star`（図鑑で👑）。
- **`mastered`（Lv3ノーヒント）の定義は変更しない。**モードC stage5 の出題プールと図鑑の金マークは従来通り mastered 基準（一度与えたバッジを取り上げない）。
- 想起チェックの上位tier: Lv3以上はパーツを共有する「似た字」を選択肢に、Lv4以上は**漢字→読みの逆引き3択**（ひらがなカード、§12.2の緩和を活用）。

### 12.6 モードC 拡張 — 拗音と「ことばモード」

- **stage 6:** 拗音（きゃ/しゅ/ちょ等）。
- **stage 7–9（ことばモード）:** 音声で単語を出題し、カードを**正しい順番にタップして綴る**（2–4文字）。正タップごとに文字音を再生し画面上部のスロットに文字が積まれる。完成で単語全体の読み上げ＋絵文字演出。**順序違いのタップは誤答扱いにしない**（そのカードの音を読み上げ、次に探す文字を案内するのみ）。stage 8=カタカナ語、stage 9=既習漢字カード混在。
- 単語データは `src/data/words.json`（約80語、促音・拗音入り、興味領域寄せ）。バリデータで文字セット整合と禁止語彙を検査。
- 単語ステージはカード最大12枚（カード縮小・最小間隔70px）。チャレンジの自己ベストはステージ別に記録（`challengeBestByStage`）。

### 12.7 モードD「ごうせいラボ」（新設・PhET型自由実験）

Build-a-Molecule の漢字版。ホームの🧪から入る。

- 解放済みパーツ（一度でも完成/発見した字のパーツ + 初期セット）が「原子」としてトレイに無限供給される。自由にドラッグして組み合わせる。
- 近接したパーツ群がレシピ（収録漢字のパーツ構成）と一致すると**結合演出 + 「はっけん！」+ 読み/成り立ち音声** → 図鑑に発見登録。
- **一致しない組み合わせは何も起きず、ゆっくり離れるだけ。**「結合しなかった」は実験結果であって失敗ではない（NO-FAIL）。音・振動・否定表示は一切出さない。
- 完成した漢字チップはタップで再分解できる（探索の可逆性）。チップ同士を重ねると**熟語合成**（例: 森+林=森林、`src/data/jukugo.json`）。
- ラボ専用の発見対象（`labOnly` エントリ、通常出題には出ない字）を含める。レシピのパーツ構成（マルチセット）はデータ全体で一意であることをバリデータで保証する（本=木+一 と 末/未 の衝突防止）。
- 「はじめての発見」はハイライト（§4.6）の選定対象に加える（初完成漢字の直後の優先度）。

### 12.8 データ互換性

- IndexedDB のストア構成・DB_VERSION は変更しない。progress への**新キー追加**と既存値の**型の拡張**（例: KanjiProgress.level 1–3 → 1–4）のみ。エクスポート/インポート（§4.5）は無修正で互換。
- 既存の `kanji.enabledIds`（保護者の出題範囲）は明示リストのため新収録字が自動では有効にならない。保護者画面に「新しい字を一括有効化」操作を設ける。
